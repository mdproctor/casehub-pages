## В§1 Introduction and Goals

### Description

`casehub-pages` is a web application framework for the CaseHub platform вҖ” a pure TypeScript runtime for composing applications from layouts, data pipelines, visualization, and forms. The framework separates data from presentation, wires them together through declarative bindings and a reactive event pipeline, and renders the result as Web Components.

Core packages: `@casehubio/pages-data` (DataSet model, reactive event engine, operations, external data extraction, WebSocket sources, JSONata), `@casehubio/pages-ui` (TypeScript DSL builders, YAML parser, component model), `@casehubio/pages-viz` (Web Component chart/table/metric wrappers around ECharts), `@casehubio/pages-component` (CSS grid layout renderer, interactive containers), `@casehubio/pages-runtime` (site orchestrator вҖ” `loadSite()` API, data pipeline, event delegation).

### Stakeholders

| Stakeholder | Interest |
|---|---|
| Application developers | Compose web applications using the TypeScript DSL |
| Host applications | Embed casehub-pages via iframe or direct integration using `loadSite()` API |
| DashBuilder users | Backward-compatible rendering of existing DashBuilder YAML |

### Quality Goals

| Goal | Scenario |
|---|---|
| DashBuilder compatibility | Existing DashBuilder YAML renders correctly without modification |
| Zero-Java runtime | Full rendering pipeline in TypeScript вҖ” no GWT compilation, no Java at runtime |
| Interactive filtering | Cross-component filtering, sorting, and pagination work end-to-end |
| Recursive composition | Any component nests inside any other вҖ” the tree is the API |
| DataвҖ“display separation | Data defined once, bound by reference; same dataset drives multiple components simultaneously |

## В§4 Solution Strategy

### Data Flow

```
TypeScript DSL (or YAML string)
    вҶ“  (DSL builders / js-yaml)
Component tree + DataSetDef[]       вҖ” layout model
    вҶ“
resolveDataSet() / WebSocketSource  вҖ” @casehubio/pages-data
    вҶ“
DataSet (columns + rows)            вҖ” reactive: mutations cascade to bound components
    вҶ“
renderNode() / renderComponent()    вҖ” @casehubio/pages-component
    вҶ“
<pages-bar-chart> etc.              вҖ” @casehubio/pages-viz Web Components
    вҶ“
pages-filter / pages-sort           вҖ” DOM custom events back to data layer
```

### Reactive Data Pipeline

The `DataSetManager` uses a unified event model вҖ” `apply(id, event)` accepts a discriminated union of four event types (snapshot, append, replace, remove). All sources вҖ” HTTP resolver, expression generator, WebSocket вҖ” produce `DataSetEvent` objects. The manager applies them uniformly, and changes cascade automatically to all bound components.

### Lazy Rendering for One-at-a-Time Containers

Tabs, pills, sidebar, carousel, and stack containers render only the active slot's children. On slot switch, the swap function tears down the old slot's DOM and renders the new slot lazily.

Key decisions:
- **Swap function as single owner** вҖ” each wire function (wireTabs, wireSidebar, wireCarousel, wireStack) builds one swap function that owns slot state, DOM lifecycle, button chrome, and event dispatch. Click handlers and `activateSlot` both delegate to it via a `slotSwapRegistry` WeakMap.
- **`slot-swap.ts` shared module** вҖ” consolidated `dispatchSlotChange` from interactive.ts and activate-slot.ts, eliminating duplication.
- **Registry eviction via `isConnected`** вҖ” `pages-slot-change` handler scans the registry and removes entries whose element is disconnected from the DOM. O(1) per entry.
- **Accordion stays eager** вҖ” all accordion sections start expanded, so lazy rendering would change UX.

### NavTree Page Deduplication

`parsePage` initially included pages both as top-level `content` slot entries AND as slot values inside navigation containers, causing N-fold DOM duplication. Fix: `collectNavTreePageNames` extracts all page names from navTree groups. These are filtered from the root `content` slot. `patchSlotReferences` recursively replaces unresolved page references in nav slots with their resolved versions.

### Cross-Filter Event Protocol

The filter flow (selector вҶ’ `pages-filter` вҶ’ listening component re-query) requires:
1. Event listener ordering вҖ” `renderComponent` must run AFTER `addEventListener` (custom elements dispatch `pages-data-request` synchronously during `connectedCallback`)
2. `@casehubio/viz` side-effect import вҖ” custom elements must be registered for `connectedCallback` to fire
3. `yamlLoad` call in the YAML string path

## В§6 Runtime View

### Application Loading

1. Host calls `loadSite(container, tree)` with a TypeScript DSL component tree (or YAML string)
2. Runtime parses input via `@casehubio/pages-ui` вҶ’ `ComponentNode` tree + `DataSetDef[]`
3. Runtime registers event listeners for `pages-data-request`, `pages-filter`, `pages-sort`, `pages-page`
4. Runtime calls `renderComponent()` вҶ’ `@casehubio/pages-component` renders CSS grid layout
5. Viz Web Components fire `pages-data-request` during `connectedCallback`
6. Runtime resolves each dataset via `@casehubio/pages-data` (HTTP, WebSocket, expression) and posts results back
7. Components render with ECharts / DOM

### Cross-Filtering

1. User clicks a data point (chart bar/slice, table cell, selector option, iframe message)
2. Emitter resolves `row` and `value` at dispatch time, emits `pages-filter` custom event with `PagesFilterApply` (discriminated union: `columnId`, `value`, `row`, `reset: false`, `group`) or `PagesFilterReset` (`columnId`, `reset: true`, `group`)
3. Runtime uses `detail.value` directly вҖ” no fallback logic or row extraction
4. If child DataScope exists and row contains `idColumn` вҶ’ record selection path; otherwise вҶ’ cross-filter path
5. Runtime re-queries listening components via the data pipeline with updated `FilterState`
6. Components re-render; emitters track selection via `_selectedValue` for toggle (click-again-to-deselect) and visual feedback (ECharts `highlight`/`downplay` for charts, `.selected` CSS for tables)

## В§7 Deployment View

Packages published to GitHub Packages (`npm.pkg.github.com`) under the `@casehubio` scope. Host applications consume packages via Quarkus Quinoa вҖ” `npm install` at build time, static serving from `META-INF/resources/` at runtime. No Node.js at runtime.

Convention: `docs/quinoa-convention.md`. Reference template: `templates/quinoa-host/`.

The examples gallery serves at `localhost:8080` via webpack-dev-server (monorepo-internal, not published).

## В§10 Architectural Decisions

Decisions not captured inline in В§4:

| Decision | Rationale |
|---|---|
| Web Components over React for viz | Components must work inside any host framework (React, vanilla, GWT iframe). Web Components are framework-agnostic. |
| Custom events over callback API | `pages-filter`, `pages-sort` decouple components from the runtime. Any host can intercept and handle events. |
| TypeScript DSL as primary API | Type-safe builders with IDE autocompletion. YAML supported for runtime-loaded pages (dynamic content). |
| DashBuilder YAML backward compat | Existing DashBuilder YAML still parses and renders. Parser handles both DashBuilder legacy and clean casehub format. |
| Unified event model for data mutations | snapshot/append/replace/remove as a discriminated union. All sources produce the same events. Enables reactive cascading. |

## В§13 Glossary

| Term | Definition |
|---|---|
| `DataSet` | Column-oriented tabular data вҖ” the wire format between data resolution and visualization |
| `DataSetLookup` | A query against a named dataset: filter вҶ’ group вҶ’ sort pipeline |
| `DataSetEvent` | Discriminated union of mutation types: snapshot, append, replace, remove |
| `ComponentNode` | A node in the parsed layout tree вҖ” carries type, props, slots (children), and optional dataset reference |
| `loadSite()` | Entry point: accepts component tree or YAML string, renders into a DOM container, returns a dispose handle |
| `NavTree` | Navigation structure: groups of named pages rendered as tabs, tree, menu, or tiles |
